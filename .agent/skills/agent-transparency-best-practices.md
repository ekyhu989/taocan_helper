## 🎯 核心原则

### 1. 绝对透明（ABSOLUTE TRANSPARENCY）

**来源**：github/awesome-copilot - Ultimate-Transparent-Thinking-Beast-Mode

**核心要求**：
- ✅ 每个关键步骤前必须声明正在执行的操作
- ✅ 展示思考过程和决策依据
- ✅ 明确标注不确定性
- ❌ 禁止黑盒执行
- ❌ 禁止只输出"✅ 完成"而不展示过程

**实施格式**：
```
🧠 THINKING: [透明推理过程]
🔧 EXECUTING: [正在执行的操作]
📋 CHECKING: [检查项] | [内容] | [结果] | [问题]
```

---

### 2. 强制逐项检查（MANDATORY SEQUENTIAL VERIFICATION）

**来源**：多项目实践总结

**核心要求**：
- ✅ 每项检查必须独立展示结果
- ✅ 禁止跳过任何检查项
- ✅ 发现问题必须立即上报
- ❌ 禁止主观判断"这个不需要检查"

**实施格式**：
```markdown
📋 检查清单执行：
1. [检查项1] ✅ 通过 - [具体内容]
2. [检查项2] ⚠️ 发现问题 - [问题描述] → 🚨 已上报
3. [检查项3] ✅ 通过 - [具体内容]
```

---

### 3. 问题上报强制机制（ISSUE ESCALATION MANDATE）

**来源**：slowmist/openclaw-security-practice-guide - behavioral self-inspection

**核心要求**：
- ✅ 遇到任何问题必须立即上报
- ✅ 包含问题描述、影响范围、建议方案
- ✅ 禁止选择性跳过或忽略
- ❌ 禁止"我觉得这个问题不大"

**上报格式**：
```
🚨 [问题类型] 
- 描述：[具体问题]
- 影响：[影响范围]
- 建议：[建议方案]
- 状态：[等待确认/已修复/需协助]
```

---

### 4. 开发日志强制输出（DEVELOPMENT LOG MANDATE）

**来源**：AI Agent Specification Template + 最佳实践

**核心要求**：
- ✅ 每完成一个功能模块必须输出开发日志
- ✅ 记录修改的文件、调用的技能、检查结果
- ✅ 作为后续审查和追溯的依据

**日志格式**：
```markdown
📝 开发日志 - [功能名称]
- 时间：[时间戳]
- 修改文件：
  - src/xxx.js（修改原因）
  - src/yyy.css（修改原因）
- 技能调用：
  - code-quality ✅ 通过
  - hallucination-prevention ✅ 通过
- 检查结果：
  - 状态流转 ✅ 完整
  - 响应式适配 ✅ 通过
  - 代码规范 ⚠️ 1个问题（已修复）
```

## 📊 验证机制

### 用户如何验证AI是否遵守？

**检查点1：执行前声明**
```
搜索对话中的"🔧 [技能名称] 执行中"
如果没有 = AI可能跳过技能
```

**检查点2：逐项检查结果**
```
搜索对话中的"检查项名称 | 检查内容 | 结果"
如果只有"✅ 完成" = AI可能没有真正检查
```

**检查点3：问题上报**
```
搜索对话中的"🚨"
如果遇到问题但没有🚨 = AI选择性跳过
```

**检查点4：开发日志**
```
搜索对话中的"📝 开发日志"
如果没有 = AI没有记录开发过程
```

## 🎓 关键教训

### 教训1：AI会懒惰
- **现象**：跳过检查、主观判断、黑盒执行
- **原因**：默认行为是"最快完成任务"
- **解决**：强制透明 + 逐项展示 + 问题上报

### 教训2：规则必须可验证
- **现象**：规则写了"必须检查"，但无法验证
- **原因**：缺少输出格式要求
- **解决**：规定具体的输出格式和关键词

### 教训3：问题上报必须强制
- **现象**：遇到问题选择性忽略
- **原因**：没有强制上报机制
- **解决**：🚨格式 + 立即上报 + 禁止跳过
