## 🚀 快速接入（3步完成）

### 方式A：自动归档（推荐 ⭐⭐⭐⭐⭐）

**适合场景：** 旧项目完整打包，让AI自动分类归档

```bash
# 一步完成：扫描 + 分析 + 归档 + 生成报告
python archive_legacy_project.py /path/to/old-project
```

**自动化流程：**
1. 🔍 扫描旧项目所有文件
2. 📊 统计项目结构（文件数、目录数、大小）
3. 🔧 检测技术栈（Node.js、Vue、微信小程序等）
4. 🔑 识别关键文件（README、配置文件、数据库脚本等）
5. 📦 自动归档到 `docs/09-归档/`
6. 📝 生成项目现状报告
7. ✅ **完成！直接与PM沟通v2.0需求**

**示例输出：**

### 方式B：手动配置

**适合场景：** 只想配置框架，不归档旧代码

### 步骤1: 复制框架到项目根目录

```bash
# 将以下文件/目录复制到你的项目根目录
.agent/              # AI Agent框架核心
.workbuddy/          # 工作记忆
AGENTS.md            # Agent协作指南
EVOLUTION_SYSTEM.md  # 自动进化系统
README.md            # 项目文档
SYSTEM_TOP.md        # 顶层规则
init_project.py      # 初始化脚本
```

**注意：** 不要覆盖项目现有文件！

### 步骤2: 运行初始化配置

```bash
python init_project.py
```

**输入项目信息：**
- 项目名称和简介
- 技术栈（后端、前端、数据库、部署平台）
- CI/CD工具
- 设计规范（颜色、字体、圆角）

**脚本会自动：**
- ✅ 替换所有角色配置中的占位符
- ✅ 生成 `.agent/project-config.json`
- ✅ 验证配置完整性

### 步骤3: 开始迭代开发

```bash
# 1. 在docs目录创建需求文档
docs/01-需求文档/v2.0-新功能需求.md

# 2. PM角色读取需求，生成规格说明书
# 3. 按照9阶段工作流执行
```

## 🎯 旧项目迭代工作流

### 第一次迭代（v2.0）

```
1. 📝 PM角色：编写v2.0需求文档
   ↓ docs/01-需求文档/v2.0-新功能.md

2. 🔍 PM角色：生成需求规格说明书
   ↓ 使用demand-spec-builder skill
   ↓ 回答office-hours 6个问题
   ↓ 完成CEO review 10维度审查

3. 🎨 Designer角色：设计新界面
   ↓ 读取现有UI设计规范
   ↓ 输出设计稿和交互说明

4. 💻 Developer角色：开发新功能
   ↓ 遵循Karpathy 4原则
   ↓ 自我代码审查

5. 🧪 QA角色：审查和测试
   ↓ 独立代码审查
   ↓ AI盲点检查
   ↓ 回归测试

6. 🚀 DevOps角色：部署上线
   ↓ 灰度发布
   ↓ 监控性能基线
```

## ⚙️ 配置调整建议

### 1. 调整角色权限（可选）

如果你的项目结构不同，可以修改角色配置：

**文件：** `.agent/roles/developer.json`

```json
{
  "permissions": {
    "allowed_directories": ["src/"],  // 修改为你的目录
    "forbidden_directories": []
  },
  "scan_patterns": [
    "src/cloudfunctions/{target}/**",  // 修改为你的扫描模式
    "src/miniapp/pages/{target}/**"
  ]
}
```

### 2. 添加现有代码上下文（推荐）

在 `.workbuddy/knowledge/` 中创建现有项目文档：

```bash
.workbuddy/knowledge/
├── api-index.md           # 现有API索引
├── database-schema.md     # 现有数据库结构
├── tech-debt.md          # 技术债务清单
└── architecture.md       # 现有架构说明
```

**作用：** AI Agent会读取这些文件，了解现有代码。

### 3. 配置角色工作流阶段

不同角色可以关注不同阶段：

**产品经理：**
```json
"workflow_stages": ["需求", "设计", "方案", "计划", "执行", "审查", "修复", "修复验证", "发布"]
```

**QA工程师（仅审查和验证）：**
```json
"workflow_stages": ["审查", "修复验证"]
```

## 🔧 常见问题

### Q1: 会覆盖现有代码吗？

**A:** 不会！框架只添加新文件和目录，不会修改现有代码。

### Q2: 需要重构现有代码吗？

**A:** 不需要！框架只管理**新功能的开发流程**，现有代码保持不变。

### Q3: 团队如何协作？

**A:** 
- PM负责需求文档
- Designer负责设计稿
- Developer负责代码实现
- QA负责质量审查
- DevOps负责部署

每个角色有独立的权限和检查清单。

### Q4: 如何处理技术债务？

**A:** 在 `docs/05-经验知识库/` 中记录技术债务，后续迭代时优先处理。

### Q5: 框架可以逐步引入吗？

**A:** 可以！建议顺序：
1. 先用PM角色写需求文档
2. 再加入Developer角色开发
3. 然后加入QA角色审查
4. 最后加入Designer和DevOps
