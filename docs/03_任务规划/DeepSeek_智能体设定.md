## 📝 工作规范

### 代码规范

1. **TypeScript 严格模式**
   - 所有代码必须有完整类型定义
   - 禁止 `any` 类型，特殊情况需注释说明
   - 接口定义优先使用 `interface` 而非 `type`
   - 泛型使用需有约束条件

2. **架构原则**
   - 单一职责原则（SRP）
   - 开闭原则（OCP）
   - 依赖倒置原则（DIP）
   - 优先组合而非继承

3. **代码组织**
   ```
   src/
   ├── components/     # 组件层
   ├── hooks/         # 逻辑复用层
   ├── stores/        # 状态管理层
   ├── utils/         # 工具函数层
   ├── types/         # 类型定义层
   └── constants/     # 常量定义层
   ```

4. **命名规范**
   - 组件: PascalCase (e.g., `SchemeGenerator`)
   - 函数: camelCase (e.g., `calculateBudget`)
   - 常量: UPPER_SNAKE_CASE (e.g., `MAX_BUDGET`)
   - 接口: PascalCase + I前缀 (e.g., `ISchemeConfig`)
   - 类型: PascalCase + T后缀 (e.g., `TSchemeType`)

### 文档规范

1. **架构文档必须包含**:
   - 设计目标与约束
   - 架构图与模块关系
   - 接口定义与数据流
   - 性能考量与优化点
   - 风险评估与备选方案

2. **代码注释要求**:
   - 公共API必须有JSDoc注释
   - 复杂算法必须有实现说明
   - 边界情况必须有处理说明
   - TODO/FIXME必须带任务编号

### 提交规范

```bash
# 格式: type(scope): V2.0-XXX 描述

# 示例
git commit -m "feat(architecture): V2.0-001 实现项目结构重构"
git commit -m "fix(types): V2.0-002 修复Scheme类型定义"
git commit -m "docs(architecture): V2.0-003 更新架构文档"
git commit -m "refactor(store): V2.0-004 优化状态管理架构"
```

## 🚫 禁止行为

### 绝对禁止

1. **破坏现有逻辑**
   - 未经评估修改V1.6稳定功能
   - 删除或修改已上线接口
   - 破坏向后兼容性

2. **技术债务累积**
   - 使用 `any` 类型逃避类型检查
   - 复制粘贴代码而不抽象
   - 硬编码魔法数字/字符串
   - 跳过错误处理

3. **性能陷阱**
   - 在渲染循环中执行复杂计算
   - 无限制的状态订阅
   - 大型组件不拆分
   - 资源重复加载

4. **安全违规**
   - 在代码中硬编码密钥
   - 不验证用户输入
   - 暴露内部实现细节
   - 不安全的eval使用

### 需要避免

1. **过度工程**
   - 为简单功能引入复杂架构
   - 过早抽象
   - 不必要的优化

2. **忽视边界情况**
   - 空值处理
   - 网络异常
   - 数据格式错误
   - 浏览器兼容性

3. **文档缺失**
   - 无架构说明
   - 无接口文档
   - 无部署文档
