## 📋 一、审查机制概述

### 1.1 核心原则

**每个Phase必须通过PM+QA双审查，方可进入下一Phase**

| 审查角色 | 审查重点 | 审查时机 | 输出物 |
|---------|---------|---------|--------|
| **PM** | 需求覆盖度、验收标准符合度、Phase级整体质量 | Phase任务完成后 | PM质量审查报告 |
| **QA** | 功能测试、回归测试、兼容性测试、性能测试 | PM审查通过后 | QA测试报告 |

### 1.2 审查流程图

```
Phase任务完成
    ↓
PM质量审查（需求覆盖度+验收标准）
    ↓
  通过？──否──→ 打回修复 → 重新审查
    ↓是
QA质量测试（功能+回归+兼容性）
    ↓
  通过？──否──→ 提交Bug → Frontend-Developer修复 → 重新测试
    ↓是
Phase审查通过 → 进入下一Phase
```

## 🔍 二、PM审查标准

### 2.1 审查维度

| 维度 | 审查内容 | 验收标准 |
|------|---------|---------|
| **需求覆盖度** | Phase任务是否100%完成 | 任务完成率100% |
| **验收标准** | 每个任务的验收标准是否全部通过 | 验收标准通过率100% |
| **Design Tokens合规** | 是否有硬编码颜色 | 全局grep 0 matches |
| **TypeScript类型** | 是否有类型错误 | `tsc --noEmit` 0 errors |
| **代码规范** | 是否有TODO/FIXME/HACK标记 | 已全部处理 |
| **文档完整性** | 需求文档、任务清单、项目看板是否更新 | 文档100%更新 |

### 2.2 PM审查流程

1. **读取Phase任务清单**（docs/04-开发任务/）
2. **逐项验证验收标准**（对照任务清单中的验收标准）
3. **执行代码合规检查**：
   ```bash
   # Design Tokens合规检查
   grep -r "#[0-9a-fA-F]\{6\}" src/ --include="*.css" --include="*.tsx"
   
   # TypeScript类型检查
   npx tsc --noEmit
   ```
4. **输出PM质量审查报告**（docs/06-质量保障/）
5. **更新项目看板和MEMORY.md**
6. **审查通过** → 通知QA介入测试
7. **审查不通过** → 打回Frontend-Developer修复

## 🧪 三、QA测试标准

### 3.1 测试维度

| 测试类型 | 测试内容 | 验收标准 |
|---------|---------|---------|
| **功能测试** | Phase新增功能 | 测试用例通过率 ≥ 95% |
| **回归测试** | Phase 1~N-1功能 | 测试用例通过率 = 100% |
| **兼容性测试** | Chrome/Edge/微信/移动端 | 主浏览器100%通过 |
| **性能测试** | 首屏加载、页面切换 | 首屏<2s，切换<500ms |

### 3.2 QA测试流程

1. **读取PM质量审查报告**（确认PM审查已通过）
2. **创建QA测试计划**（docs/06-质量保障/）
3. **执行功能测试**（Phase新增功能）
4. **执行回归测试**（历史功能未受影响）
5. **执行兼容性测试**（Chrome/Edge/微信/移动端）
6. **输出QA测试报告**（docs/06-质量保障/）
7. **测试通过** → Phase审查完成，进入下一Phase
8. **测试不通过** → 提交Bug清单 → Frontend-Developer修复 → 重新测试

## ⚠️ 五、注意事项

### 5.1 PM审查边界

- ✅ PM允许：读取代码、分析问题、输出审查报告、打回修复
- ❌ PM严禁：修改src/目录下任何文件、代替其他角色修复代码

### 5.2 QA测试边界

- ✅ QA允许：执行测试、提交Bug、输出测试报告
- ❌ QA严禁：修改src/目录下任何文件、跳过PM审查直接测试

### 5.3 Frontend-Developer修复边界

- ✅ 允许：修复PM/QA提交的问题、提交修复记录
- ❌ 严禁：跳过审查直接提交、修改未授权的文件

## 🔄 七、审查流转规则

### 7.1 允许流转

- ✅ PM审查通过 → QA测试
- ✅ QA测试通过 → 进入下一Phase
- ✅ QA测试不通过 → Frontend-Developer修复 → QA重新测试

### 7.2 禁止流转

- ❌ PM审查不通过 → 进入QA测试（必须修复后重新审查）
- ❌ QA测试不通过 → 进入下一Phase（必须修复后重新测试）
- ❌ 跳过PM审查 → 直接进入QA测试
- ❌ 跳过QA测试 → 直接进入下一Phase
