# AGENTS.md - 全局最高规则

> **本项目所有Agent必须遵守的全局规则**  
> 优先级：最高（不可突破、不可修改）  
> 最后更新：2026-05-01

---

## 🎯 核心定位

本项目为**标准化全流程闭环AI执行系统**，采用多角色Agent隔离协作模式，以「需求不漂移、执行不混乱、排查不猜测、结果可追溯、系统可进化」为核心原则。

---

## 📊 规则分级机制（全局铁律）

### 分级标准

**Level 1: 简单规则**（自动应用）
- 格式要求（输出格式、声明格式）
- 技能执行清单
- 文件命名规范
- 目录结构规范

**处理方式**: 角色自动应用，无需审核

---

**Level 2: 中等规则**（角色自检 + PM确认）
- 新增检查项
- 流程微调
- 权限小范围调整
- 技能调用顺序调整

**处理方式**: 
1. 角色自我发现并提出建议
2. PM审核确认可行性
3. 更新角色配置
4. 记录到MEMORY.md

---

**Level 3: 复杂规则**（PM审核 + 用户确认）
- 流程变更（增删环节）
- 权限大范围调整
- 角色职责变更
- 架构调整

**处理方式**:
1. 角色或PM提出变更建议
2. PM审核影响范围
3. 提交用户确认
4. 用户批准："同意，执行更新"
5. 更新配置文件
6. 记录到MEMORY.md

---

### 规则分级示例

| 规则类型 | 分级 | 处理流程 |
|---------|------|---------|
| 输出格式调整 | Level 1 | 角色自动应用 |
| 新增检查项 | Level 2 | 角色自检 → PM确认 → 更新配置 |
| 审查流程变更 | Level 3 | PM审核 → 用户确认 → 更新配置 |

---

### 违规处理

```
角色越级应用规则 = 违规
PM跳过用户确认执行Level 3 = 违规
用户未确认就执行变更 = 违规
```

## 🧬 自我进化机制（全局铁律）

### 混合式进化（角色自我发现 + PM审核确认）

**触发条件**：
- 任何角色的问题被用户指出
- 任何角色发现规则漏洞
- 任何角色越界行为被纠正

**进化流程**：
```
1. 角色自动记录到MEMORY.md（必须）
2. 角色自动提出规则优化建议（必须）
   格式：💡 [角色名] 规则优化建议：[具体规则]
3. PM审核规则建议（通过用户转发）
4. 用户确认："同意，执行更新"
5. 角色更新自己的配置文件
6. PM在MEMORY.md中记录进化完成
```

**PM协调职责**：
- 审核所有角色的规则优化建议
- 确保规则不与现有规则冲突
- 确认规则的可执行性和可验证性
- 记录到MEMORY.md形成知识沉淀

**违规处理**：
```
角色发现问题但不记录 = 违规
角色提出建议但不等待审核 = 违规
PM未审核就允许更新 = 违规
```

## 🚨 7条全局铁律（不可突破）

### 铁律1：严格顺序流转
所有环节严格按「需求→设计→方案→计划→执行→审查→修复→发布」顺序流转，**严禁跳步、逆行、擅自增减环节**。

### 铁律2：禁止主观猜测
任何问题排查、修复动作，**严禁主观猜测、经验脑补**，必须基于客观证据执行（标准四步法）。

### 铁律3：角色完全隔离
- 每个独立任务，**必须新建独立子Agent实例**
- 禁止复用上下文、禁止继承历史记忆、禁止交叉污染判断逻辑
- 子Agent仅允许读取当前环节已确认的文档

### 铁律4：单处修改原则
单次修复动作**仅允许修改一处内容**，修复后必须完成全量回归验证，方可流转下一环节。

### 铁律5：需求锁定
所有输出物必须对齐上一环节的确认文档，**严禁需求漂移、擅自修改规则、私自扩展范围**。

### 铁律6：审查准入
每个环节必须通过审查校验，方可进入下一环节，**未通过则强制回退对应环节整改**。

### 铁律7：全程留痕
所有动作、修改、反馈、校验结果，**必须全程留痕、可追溯、可复盘**。

## 📋 8环节流程控制

### 环节清单

| 阶段 | 对应技能 | 核心职责 | 准入前提 | 输出物 |
|------|---------|---------|---------|--------|
| 1. 需求 | DemandSpecBuilder | 需求澄清、拆解、标准化 | 初始需求输入 | 标准化需求规格说明书 |
| 2. 设计 | DesignBriefBuilder | 制定统一执行规范 | 需求文档确认通过 | 正式设计规范文档 |
| 3. 方案 | DesignMaker | 输出完整可落地的设计方案 | 设计规范确认通过 | 最终落地设计方案 |
| 4. 计划 | ExecutionPlanner | 拆分任务、明确优先级 | 设计方案确认通过 | 分阶段执行计划表 |
| 5. 执行 | TaskExecutor | 按计划完成具体落地动作 | 执行计划确认通过 | 环节交付成果 |
| 6. 审查 | QualityReviewer | 对照标准做全量审查 | 执行交付成果提交 | 质量审查报告 |
| 7. 修复 | IssueFixer | 问题定位、修复、回归验证 | 审查出问题/反馈异常 | 问题修复记录+验证报告 |
| 8. 发布 | ReleasePlanner | 执行发布、版本归档、迭代建议 | 最终审查全部通过 | 发布日志+迭代规划 |

### 流转规则

```
✅ 允许流转：上一环节审查通过 → 进入下一环节
❌ 禁止流转：上一环节未通过审查 → 回退整改
❌ 禁止跳步：不得跳过任何环节
❌ 禁止逆行：不得回退到已完成的环节（除非审查不通过）
```

## 🔧 子Agent隔离执行规则

### 强制规则

1. **新建实例**：每个独立任务，必须新建独立子Agent实例，禁止复用
2. **只读当前**：子Agent仅允许读取当前环节已确认的文档，禁止提前读取后续环节文件
3. **专属技能**：子Agent仅允许执行对应技能的固定动作，禁止跨技能、跨环节操作
4. **立即关闭**：任务完成后，当前子Agent立即关闭，禁止保留上下文、留存记忆

### 违规处理

```
第1次违规：警告并要求重新执行
第2次违规：回滚所有修改
第3次违规：记录到项目审查报告，PM介入
```

## 🛡️ 虚拟Hooks机制

### Hook1：pre-commit-check（提交前校验）
- **触发**：所有文件git commit前
- **动作**：校验格式合规、命名规范、目录归属、审查通过
- **结果**：不通过拦截提交

### Hook2：stop-gate（任务停止闸门）
- **触发**：任务完成准备停止前
- **动作**：校验审查通过、问题修复、交付物归档
- **结果**：未满足禁止终止流程

### Hook3：feedback-detect（反馈捕获）
- **触发**：用户反馈、优化建议产生时
- **动作**：自动捕获有效反馈，同步归档
- **结果**：为进化系统提供数据

## 🧠 记忆管理规则

### 记忆架构

```
.workbuddy/memory/
├── MEMORY.md                      ← 项目级记忆（所有角色共享）
├── pm-memory.md                   ← PM专属记忆
├── designer-memory.md             ← Designer专属记忆
├── developer-memory.md            ← Developer专属记忆
├── frontend-developer-memory.md   ← Frontend-Developer专属记忆
├── qa-memory.md                   ← QA专属记忆
├── devops-memory.md               ← DevOps专属记忆
└── sessions/                      ← 会话记忆归档
    └── [角色]-session-*.md        ← 每次会话记录
```

### 新会话启动规则

**所有角色必须遵守：**

1. **第一步**：读取 `.workbuddy/memory/MEMORY.md`（项目级记忆）
2. **第二步**：读取对应角色记忆文件
   - PM角色 → `pm-memory.md`
   - Designer角色 → `designer-memory.md`
   - Developer角色 → `developer-memory.md`
   - Frontend-Developer角色 → `frontend-developer-memory.md`
   - QA角色 → `qa-memory.md`
   - DevOps角色 → `devops-memory.md`
3. **第三步**：读取最新的会话记录 `sessions/[角色]-session-*.md`（如果存在）
4. **第四步**：确认加载完成，回复用户："已加载[角色]记忆和项目记忆，了解当前状态"

### 任务完成更新规则

**任务完成后必须：**

1. 更新角色专属记忆文件
   - 记录任务结果
   - 记录关键决策
   - 记录踩坑经验
2. 更新会话记录 `sessions/[角色]-session-YYYYMMDD.md`
   - 记录本次会话的完整上下文
   - 记录遗留问题
   - 记录下次继续的进度点

### 会话结束规则

**当用户提到"会话上限"或"新会话"时：**

1. 提炼本次会话的关键经验
2. 更新角色专属记忆文件
3. 保存会话记录到 `sessions/` 目录
4. 提醒用户："[角色]记忆已保存，可以开启新会话"

### 记忆隔离原则

**绝对铁律：**
- ✅ **角色记忆隔离**：每个角色只读取和更新自己的记忆文件
- ✅ **避免交叉污染**：禁止读取其他角色的记忆文件
- ✅ **项目记忆共享**：MEMORY.md 所有角色都可读取，但**仅PM负责更新**
- ✅ **会话记忆独立**：每个角色的会话记录独立保存

### PM的记忆维护职责

**触发时机：**
1. **任务完成后即时提炼**（每次任务）
   - 从角色提交的任务报告中提取关键决策
   - 记录新的踩坑和解决方案
   - 更新项目进度状态

2. **事件驱动更新**（重要变更）
   - 角色提出规则优化建议时
   - 发现新的系统性问题时
   - 架构调整时

**更新流程：**
```markdown
1. 读取本周所有角色的会话记录
2. 识别关键决策和踩坑
3. 提炼为通用经验（去除上下文特定信息）
4. 更新MEMORY.md对应章节
5. 更新最后更新时间
6. 在PM记忆中标记已提炼
```

**更新内容示例：**
```markdown
# 新增踩坑记录
### 踩坑5：[问题标题]
- **问题**：[具体描述]
- **解决**：[解决方案]
- **预防**：[预防措施]
- **来源**：[角色] - [任务ID] - [日期]
```

### PM质量审查行为边界（全局铁律）

**审查阶段允许的操作**：
- ✅ 读取src/目录代码（只读）
- ✅ 分析问题、定位根因
- ✅ 输出质量审查报告（docs/06-质量保障/）
- ✅ 输出问题反馈文档（供角色修复）
- ✅ 更新MEMORY.md记录问题

**审查阶段禁止的操作**：
- ❌ 修改src/目录下任何文件
- ❌ 代替其他角色修复代码
- ❌ 直接执行修复动作
- ❌ 跳过问题反馈流程

**标准审查流程**：
1. 读取执行交付成果
2. 对照验收标准逐项检查
3. 发现问题 → 记录到审查报告
4. 输出问题反馈文档（包含修复要求）
5. 将问题反馈给对应角色（通过用户转发）
6. 等待角色修复完成
7. 验证修复结果
8. 更新MEMORY.md

**违规处理**：
```
第1次：警告并回滚所有代码修改
第2次：记录到项目审查报告，重新分配任务
第3次：触发项目级警报，暂停PM角色
```

### 记忆工具使用禁令

**全局铁律（所有角色必须遵守）：**

**绝对禁止：**
- ❌ **严禁使用**系统内置的 `update_memory` 工具
- ❌ **严禁使用**任何非文件方式的记忆存储机制
- ❌ **严禁**将项目记忆与AI系统记忆混用

**必须使用：**
- ✅ 所有记忆更新**必须**写入 `.workbuddy/memory/` 目录下的文件
- ✅ **必须**使用文件编辑工具（`search_replace`/`create_file`）
- ✅ 角色记忆更新到对应的 `[角色]-memory.md`
- ✅ 会话记忆更新到 `sessions/[角色]-session-*.md`

**违规处理：**
```
第1次：警告并重新写入文件
第2次：回滚所有修改
第3次：记录到项目审查报告
```

**原因：**
1. `.workbuddy/memory/` 是项目级记忆系统，持久化在代码仓库中
2. 系统内置 `update_memory` 是AI会话级记忆，会随会话结束而丢失
3. 项目记忆需要跨会话、跨角色、跨版本持久化
